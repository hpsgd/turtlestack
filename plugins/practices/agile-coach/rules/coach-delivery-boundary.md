# Agile coach / delivery manager boundary

When both an agile coach and a delivery manager are active, route work by one rule: the **coach owns the team** (internal process), the **delivery manager owns the delivery** (external coordination). The coach changes how the team works; the delivery manager shepherds the work through the organisation. Most routing confusion between the two resolves by asking "is this inside the team boundary or outside it?"

## Default ownership

| Activity | Owner | Why |
|---|---|---|
| Sprint planning facilitation | Agile coach | Team event, internal |
| Daily Scrum | Team (coach observes) | Developer-owned per the Scrum Guide |
| Refinement | Product owner runs, coach observes | PO owns clarity; coach watches INVEST and capacity |
| Sprint review | Product manager runs, coach observes | Stakeholder-facing inspection of the increment |
| Retrospective | Agile coach facilitates | Team process inspection |
| Backlog grooming | Product owner | Execution-side ownership |
| Working agreements | Agile coach | Team norm artifact |
| RAID log | Delivery manager | Governance artifact |
| Dependency board | Delivery manager | Cross-team coordination |
| Status report | Delivery manager | Stakeholder communication |
| Release readiness | DM coordinates, release-manager executes | DM shepherds, RM gates |
| Definition of Done | Team authors, coach facilitates, DM checks at release | Team contract, gate at release |
| Impediments inside the team | Agile coach | Team-internal |
| Impediments outside the team | Delivery manager | Organisational |
| Service assessment readiness | Delivery manager | Programme-level gate |
| Reference-class forecasting | Delivery manager | Commitment discipline |
| Psychological safety scan | Agile coach | Team health |
| Flow metrics, WIP limits | Coach coaches, DM reads | Coach changes the practice; DM uses the data for status |

## The two seams worth naming

- **Flow metrics.** The coach and DM both touch them, but differently: the coach coaches the team to read and improve its own flow; the DM reads the same data for status reporting. The coach changes the practice; the DM never does.
- **Definition of Done.** The team authors it, the coach facilitates that authoring, and the DM checks it at the release gate. Three roles, one artifact, no overlap — each touches it at a different moment.

## Boundaries that hold

- The delivery manager does not facilitate retrospectives. Retros stay with the coach, exclusively. If a project runs a DM without a coach, retros go uncovered — that is a project setup gap, not something the DM papers over.
- The coach does not own RAID, dependencies, or status. Team health is not delivery shepherding.
- Neither absorbs the other when one is missing. A missing role is a missing role — name the gap rather than stretching the present role across both jobs.
